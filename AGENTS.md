# Руководство для контрибьютеров Digger

Этот документ содержит информацию для разработчиков, желающих внести свой вклад в проект Digger - Java-клон классической игры Digger.

## Структура проекта и организация модулей

Проект имеет стандартную Maven-структуру:
```
src/main/java/org/digger/app/   # Основной исходный код игры
    Digger.java                  # Главный класс игры
    Main.java                    # Точка входа
    cgagrafx.java                # Графический модуль
    Pc.java                      # Логика игрока
    Scores.java                  # Система очков
    BagState.java, GameState.java, MonsterState.java # Data classes for game state
    alpha.java, Bags.java, Drawing.java, Input.java, Monster.java, Sound.java, Sprite.java # Дополнительные компоненты
pom.xml                         # Конфигурация Maven
README.md                       # Описание проекта
digger.ini                      # Конфигурация игры
```

## Команды сборки, тестирования и разработки

### Сборка проекта
```bash
# Сборка проекта с Maven
mvn clean compile

# Создание исполняемого JAR
mvn clean package

# Запуск скомпилированной игры
java -jar target/digger-app-1.0.0.jar
```

### Разработка и отладка
```bash
# Компиляция вручную (альтернатива Maven)
javac -d target/classes src/main/java/org/digger/app/*.java

# Запуск из скомпилированных классов
java -cp target/classes org.digger.app.Digger

# Проверка зависимостей
mvn dependency:tree
```

## Стиль кода и правила именования

### Форматирование кода
- Используйте Java 17 как целевую версию
- Соблюдайте стандартные соглашения Java по именованию:
  - Классы: `PascalCase` (Digger, Main, Scores)
  - Методы и переменные: `camelCase` (getScore, movePlayer)
  - Константы: `UPPER_SNAKE_CASE` (MAX_SCORE, GAME_SPEED)
- Отступы: 4 пробела (не табуляция)
- Длина строки: рекомендуется не более 120 символов

### Организация кода
- Основная логика игры находится в классе `Digger.java`
- Графический рендеринг - в `cgagrafx.java`
- Управление игроком - в `Pc.java`
- Звуки и графика хранятся в соответствующих классах

### Тестирование
- В настоящее время проект не включает юнит-тесты
- При добавлении новой функциональности рекомендуется создавать соответствующие тесты
- Тесты должны располагаться в `src/test/java/org/digger/app/`

## Рекомендации по VCS: коммиты и пулл реквесты

### Соглашения о коммитах
Используйте семантические сообщения коммитов:
```
feat: добавление новой функциональности
fix: исправление ошибки
docs: обновление документации
style: изменения форматирования (не влияющие на логику)
refactor: рефакторинг кода без изменения функциональности
test: добавление или изменение тестов
chore: обновление зависимостей, конфигураций
```

Примеры:
- `feat: добавить систему сохранения результатов`
- `fix: исправить обработку столкновений с монстрами`
- `docs: обновить AGENTS.md с новыми командами`

### Требования к пулл реквестам
1. **Описание**: Четко опишите изменения и их цель
2. **Связанные задачи**: Укажите ссылки на соответствующие issues
3. **Тестирование**: Убедитесь, что изменения не ломают существующую функциональность
4. **Согласованность**: Проверьте соответствие стилю кода проекта
5. **Минимальные изменения**: Делайте PR максимально сфокусированными

## Архитектура и ключевые компоненты

### Основные модули игры:
1. **Игровой движок** (`Digger.java`) - управление игровым циклом и состоянием
2. **Графический рендерер** (`cgagrafx.java`) - отрисовка игрового поля и объектов
3. **Система ввода** (`Input.java`) - обработка пользовательского ввода
4. **Логика игрока** (`Pc.java`) - управление персонажем Digger
5. **Система очков** (`Scores.java`) - учет и отображение результатов

### Конфигурация:
- `digger.ini` - файл настроек игры (скорость, масштаб)
- `pom.xml` - конфигурация сборки Maven

## Советы по разработке

### Для новых участников:
1. Начните с изучения `Digger.java` - основного класса игры
2. Изучите структуру игрового цикла в методе `run()`
3. Поэкспериментируйте с настройками в `digger.ini`
4. Обратите внимание на AWT-графику в `cgagrafx.java`

### Для опытных разработчиков:
1. Рассмотрите возможность рефакторинга графического модуля
2. Добавьте юнит-тесты для критических компонентов
3. Улучшите обработку ошибок и логирование
4. Оптимизируйте производительность рендеринга

## Контакты и поддержка
